复现流水线全景

一份研报从 PDF 到最终报告要过 12 个阶段。每个阶段结束时由机器门禁（check_gates）逐项核验产物，PASS 才放行，FAIL 按预设分支修复重来——口头声称完成无效，判定只认文件和数字。

支持五类研报——选股因子（factor）、择时（timing）、资产配置（allocation）、固定收益（fixed_income）、机器学习（ml）——并按 easy / medium / hard 三档难度自动分诊，差异化投入：难度越高，审计越密、迭代额度越多、无依赖模块并行实现。

安装与首次配置

环境要求

  • uv（硬前置）：所有工具经 uv run 调用；依赖缺失时在工作目录执行 uv sync。
  • Claude 订阅档位：7 个子 agent 中 6 个使用 opus，建议 Max 档位。可自行编辑安装副本 agents/*.md 的 model 字段下调（复现质量自负）。
  • codex CLI（软前置）：外部三审查点使用。未安装或额度用完都不断链——自动降级为 Claude 替身盲审（同一审查任务、独立上下文、意见照样逐条回应闭环），替身也不可行才标记跳过；发生失败性降级时最终报告如实标注，可信度评级封顶 B。装好 / 额度恢复后无需重新配置，下次运行自动启用。

三步开始

  1. 安装插件（两步）：/plugin marketplace add https://github.com/Hedicx2020/ghquant_plugin 注册源，然后 /plugin install quant-report-reproduce@hedi-quant 安装（重启会话生效）
  2. 进入你自己的工作目录（任意空目录），运行 /reproduce setup——一次问卷收五项配置，随后自动落地：.reproduce.json 配置、templates/（数据目录模板与达标标准）、common/（内置回测库）、pyproject.toml、目录树，并检测 uv / Python 依赖 / codex。
  3. 维护 templates/data_catalog.md（见下），把 PDF 放进 reports/，运行 /reproduce reports/xxx.pdf。

setup 幂等可重跑：已有配置默认保留，你改过的模板文件永不被覆盖（插件侧更新只列清单提示，不动你的文件）。

配置详解（问卷六项 + 配置文件项）

配置项字段默认含义与影响
数据路径data_root~/local_data本地 parquet 数据库根目录。common/data_loader.py 从这里读行情、财务等数据。
执行模式default_modeautoauto：全自动优先——研报有歧义时按最合理假设放行并高亮登记，跑完统一人工 review，可事后定向重跑。interactive：遇关键（blocking 级）歧义暂停，人工裁决后继续。
最大迭代次数default_max_iter按难度自动指标未达标时自动迭代修正的轮数上限。默认 easy 3 / medium 5 / hard 6，可用 1–10 的整数全局覆盖。轮次越多复现越充分，耗时与成本越高。
回测框架backtest_framework内置 common/可指定你自有回测框架的目录路径（工具校验存在性）。指定后，复现代码的回测执行层优先调用你的框架，内置库仅补缺口；输出产物与门禁要求不变。
偏差容忍default_max_rel_dev按内置标准你能接受的复现值与原报告的偏差。默认走 standards.json 分类型精细容差；自定义（0.5%–50%）后，所有按相对偏差判定的指标统一用该容忍度，达标门禁与最终报告一致生效。
经济模式economy关闭开启后机械性角色（提取 / 验证 / 样本外分析）派发降为 sonnet，token 约降三成；写码 / 审计 / 归因等质量敏感角色保持 opus 不受影响。
外审档位audit_levelstrict配置文件项（不进问卷）。standard 时规格/代码外审改为触发式（关键歧义、首次类型、hard、ml 才跑，跳过时明示留档），结果外审任何档位必跑——反虚报是最后防线。
偏差是「相对偏差」：判定式为 |复现值 − 研报值| ÷ |研报值|。研报年化 10%、容忍 5% ⇒ 允许区间 9.5% ~ 10.5%（不是 5%~15%）。研报值越小同样容忍越苛刻，所以近零指标（如因子 IC）在内置标准里配有绝对偏差兜底（abs_eps），自定义容忍不改变这些特殊判定的语义。

优先级固定：命令行显式参数 > .reproduce.json > 内置默认。配置文件可直接编辑，或重跑 /reproduce setup 选「重新配置」。

数据目录维护（关键一步）

templates/data_catalog.md 是分诊阶段判断「研报所需数据是否可得」的唯一依据：写漏会被误判为数据缺失（对应支线降级不做），写多会引发运行期报错。建议每张表一节，写明文件名、关键列、时间范围、口径注意事项——尤其财务数据必须按披露日（info_publ_date）对齐，系统的代码审计会逐点核查未来函数。

运作方式

主会话是编排大脑：只派发子 agent、审门禁、调 codex 外审、记状态账。所有内容产物（提取规格、复现计划、代码、验证结论、最终报告）一律出自子 agent；所有达标判定一律出自门禁工具按标准重算。这个分离是防偷懒设计的根基——生产者、验证者、判定者三方互不为师。

六条硬规则

  1. 门禁即代码：每个阶段结束必须运行 check_gates 并原样呈现输出；FAIL 禁止进入下一阶段，口头声称通过无效。
  2. 编排与生产分离：主会话严禁亲自撰写规格、计划、代码、验证结论、最终报告。
  3. 产物合同逐一点收：agent 返回后逐一核验其输出文件是否真的存在——不看自述，看文件。
  4. 状态先行：阶段开始/结束先写状态文件（state.json），杜绝「跑完再补账」。
  5. 审计逐条回应：每条审计意见编号登记、逐条处置（采纳修复/说理拒绝/搁置备查），critical 必须修复并复审通过。
  6. 达标判定唯一出口：pass / partial 只能由门禁工具按 standards.json（叠加你的偏差容忍）重算得出。

七个子 agent：各司其职、互相制衡

quant-extractoropus
研报忠实提取者

只回答「研报说了什么」：产出规格书、覆盖矩阵、歧义清单三件套，每条要素带页码引用。只陈述不设计，防止提取阶段掺入主观发挥。

quant-planneropus
复现设计者

只读规格书不读 PDF（信息隔离），做类型/难度分诊、数据可得性映射、里程碑拆分、歧义裁决，产出复现计划与假设登记簿。

quant-coderopus
策略实现者

按计划切片实现策略代码并回填矩阵实现位置。只许冒烟运行，不得自行宣布验证结论——写代码的人不给自己打分。

quant-verifieropus
运行验证者

亲自跑 main.py、逐项对数产出对比表与图表、随跑随记证据链；触发时做扰动测试（改数据截止日重跑，指标必须变化——抓硬编码造假）。不采信 coder 声明。

quant-auditoropus
对照审计者（三模式）

spec / code / result 三种模式：查提取有没有漏、实现是否忠实原文、结论有没有编。只拿文件不拿被审者自述，关键数值独立重算。

quant-diagnoseropus
迭代归因者

指标超差时读全部历史轮次做根因归因、下修正指令。防兜圈：同一指标连续多轮无改善触发红线，结论三选一——继续修 / 接受局部达标 / 上报阻塞。

quant-oos-analystopus
样本外表现分析者

复现达标后，把策略原样延伸到研报回测区间之后的数据上运行，评估效应延续/衰减/失效。红线：严禁改动策略逻辑与参数——任何为改善样本外表现的调整都是数据窥探。

quant-reportersonnet
报告汇总者

把全部管线产物汇总为最终报告（含可信度评级与全量假设登记簿）。只汇总不新增结论、不重算指标——数字一律引用验证产物。

硬约束：8 个子 agent 不得再派 agent、不碰状态文件；并行只由主会话在预设的三处发起（hard 难度无依赖里程碑多 coder、规格审计双通道、代码审计与验证可并发）。

三层审计体系：防偷懒、防遗漏、防虚报

层执行者防什么手段
外审三审查点codex CLI（read-only 沙箱，全难度必跑）提取漏抄 / 实现走样 / 结论造假规格审计（盲提取对照：codex 不看规格书独立抄一遍研报数值再逐格 diff）、代码审计（公式对照 + 未来函数）、结果审计（数字与产物一致性）。每条意见编号进回应总表，critical 必须修复 + 缩减复审通过。
内审quant-auditor同上，独立视角三模式对照审计；hard 难度逐里程碑插入实现忠实性审计；ml 类无视难度必审（训练泄漏数值验证抓不到）。
反虚报核查quant-auditor（result 模式）+ 机器检查项「声称做了实际没做」K 系列（K1–K8）：结果过于完美嫌疑、声称通过实际未改等；E 系列（E1–E6）证据链：运行日志、产物新鲜度、通过数亲自重数、图表逐张数值级核验。

迭代引擎与人工闸门

首轮验证如有指标超差，进入迭代：每轮由 diagnoser 归因 → coder 定向修改 → verifier 全量重跑对数，三件套（诊断 / 修改记录 / 复验对比）落档留痕。迭代受两道约束：

  • 轮数上限：难度默认 3/5/6 轮，或你在 setup 配置的全局值；
  • 防兜圈规则：同一指标连续三轮无实质改善触发红线，diagnoser 必须改判 stop_partial（接受局部达标，残余项逐条归因）或 blocked（上报阻塞），不许无限重试。

整条流水线只有两个人工闸门，任何自动驱动（包括 /goal 无人值守）都不得冲过：

  • paused_blocked 遇到必须人工裁决的问题（如关键公式原文缺失、数据无法获取）时挂起，呈候选方案等你拍板；
  • awaiting_review 最终报告完成后等你评审——accept 收尾或 revise 定向重跑，系统绝不自行验收。

达标判定与可信度评级

每个量化指标按「研报值 vs 复现值」的相对偏差判定，容差来自 templates/standards.json（分类型精细配置，落地在你目录里可手工细调）叠加你的全局偏差容忍。特殊语义独立判定：近零指标走绝对偏差（abs_eps）、方向类指标查同号、成交笔数类查数量级、定性结论（如分组单调性）按预期字符串核对。

最终报告给出 A / B / C 可信度评级：

评级判据
A覆盖率 ≥ 90%（核心要素 100%）＋ 三外审通过 ＋ 零 critical 遗留
B核心要素覆盖 100%，但支线有缺，或外审有失败性降级（codex 不可用改由 Claude 替身盲审，或替身也不可行被跳过）；外审档位 standard 未触发的配置性跳过不算降级、不封顶
C存在核心要素未复现或 major 意见遗留未回应——报告结论区显著提示成因
评级 C 不一定意味着复现质量差：数据客观缺失（如某指数系列本地零覆盖，经穷尽核实）导致核心支线不可行时，字面判据同样触发 C——报告会写明成因与主线结论是否受影响，残余偏差 100% 归因后才能出报告。

两种复现模式：严格 vs 实验

默认为严格模式（数值对齐原文）。复现海外报告/论文而本地没有原文市场数据时，用实验模式做市场迁移：/reproduce reports/us_paper.pdf --experimental。

维度严格模式 strict（默认）实验模式 experimental（市场迁移）
目标复现值与原文数值在容差内对齐验证「该方法在迁移市场是否成立」——效应存在与否都是有效结论
数据按原文数据源（本地缺失→降级或阻塞）等价替代并逐条留痕：美国 CPI → 中国 CPI、SPX → 沪深300……全部登记假设登记簿（market-transplant 条目）
达标判定逐指标容差重算（相对偏差/绝对偏差/同号/量级），超差触发迭代修正数值对齐判定整体豁免——只核每个原文指标有迁移市场产出；原文值仅作对照参考；迭代阶段天然跳过
不变的部分提取忠实性、代码审计（公式对照/未来函数）、反虚报核查（数字与产物一致、结论不夸大）、样本外分析、人工评审闸门——两种模式完全一致
明确提示—强制：结果页顶部醒目声明框 + 徽章，最终报告必含「市场迁移声明」章节（门禁核验，缺失过不了报告关）
切换权只在你手里：未加 flag 而规划发现原文市场整体不可得 → 系统挂起呈报「建议实验模式」等你裁决，agent 无权自行切换

三个典型场景

场景一：新起复现 + 无人值守。起跑后系统会打印一条可复制的 /goal 命令，粘贴即启用跨会话自主推进——推进到四个终态之一（等待评审 / 完成 / 局部完成 / 阻塞挂起）自动停下汇报。目标措辞被刻意设计成「到达终态即停」而非「复现成功」，防止驱动层为达标冲撞人工闸门。

/reproduce reports/treasury_timing.pdf
# → init 完成后照提示粘贴 /goal 命令，然后可以去干别的

场景二：评审后改假设。报告完成后系统挂起等你评审。假设登记簿里每条自动假设（尤其高亮的 major-auto 级）都附了一条现成的 revise 命令——不认同就复制改口径：

/reproduce revise treasury_timing --assumption AS4 "涨跌幅采用结算价口径而非收盘价"
# → planner 改假设 → 按影响面只重跑受影响链路 → 报告增量更新 → 回到评审

场景三：断点续跑。会话中断、机器重启都不丢进度——状态文件是唯一真相源：

/reproduce status            # 案例一览表：编号 | id | 状态 | 阶段
/reproduce continue r3       # 编号缩写即可，不必敲全名

常见问题

我怎么知道每个案例的 id？

三个途径：① 起跑时终端会明示「本案例编号 rNNN（report_id=rNNN_短名）」——新案例的 id 由系统自动编号，形如 r001_style_factor（三位顺序号 + 看名知义的语义短名，短名取 PDF 文件名或研报标题）；② 任何时候跑无参 /reproduce status，得到按编号排序的案例一览表（编号、id、类型、状态、当前阶段）；③ 目录名即 id：workspace/ 下每个子目录对应一个案例。日常引用不必敲全名——continue/status/revise/accept 都接受编号缩写（r3、r003、3 等价）或唯一前缀，输入有歧义时系统会列出候选让你挑。想用自己的命名跑 --id 自定义名 即可（不强加编号）。

没装 codex CLI，或跑到一半额度用完了，会怎样？

都不会断。外审走两级降级链：一级降级——派一个全新的 Claude 子 agent 作"外审替身"，执行与 codex 完全相同的审查任务（同一份审查指令、只读原始材料、禁止接触流水线过程叙事），意见照样逐条编号进回应总表、critical 照样必须修复闭环；二级降级——替身也不可行（罕见）才把该审查点标记跳过并在报告显著标注。细节上做了三处省时设计：未安装时零成本直接降级（不做无谓调用，且同一案例后续审查点不再反复探测）；额度/认证类报错（usage limit、429 等）直接降级（不做无意义的缩减重试）；临时性失败（超时/网络）才重试一次。

代价如实：替身与被审代码同为 Claude 家族，少了异构引擎交叉验证这一层，所以发生任一失败性降级时可信度评级封顶 B（替身保住的是审查覆盖，不是评级）。结果审计（反虚报最后防线）在任何配置下都必须至少走到替身，不允许直接跳过。装好 codex / 额度恢复后无需重新配置，下次运行自动用回 codex。

偏差容忍 5% 到底是什么意思？研报年化 10%，容到 10.5% 还是 15%？

是 10.5%。判定用相对偏差：|复现值 − 研报值| ÷ |研报值| ≤ 容忍度。研报 10%、容忍 5% ⇒ 区间 9.5%~10.5%；你自定义 10% ⇒ 9%~11%。

想要「±5 个百分点」这种绝对偏差语义的话，编辑你目录里的 templates/standards.json 给特定指标配 abs_eps。

报告评级 C，是复现失败了吗？

不一定。C 的字面判据是「存在核心要素未复现」——常见成因是研报依赖的某类数据你本地没有（系统会穷尽核实后正式降级，理由与证据落档）。看报告结论区：会写明 C 的成因、报告落脚点结论是否受影响、残余偏差的逐条归因。真正要警惕的是无归因的偏差——而那过不了结果审计门禁。

样本外分析是什么时候做的？结论怎么读？

复现达标（pass 或 partial）后自动触发：把复现好的策略原样（零参数改动）延伸到研报回测区间之后的数据上运行，产出样本内外指标对比、净值延伸图与四选一结论——延续 / 衰减 / 失效 / 样本不足（样本外不足 60 个交易日时强制标注统计意义有限）。判读阈值写在 oos_report.md 里。门禁会核验样本外区间与研报区间零重叠，防止样本内数据冒充样本外。若你本地数据还没超出研报区间，该阶段自动跳过并留痕。

研报某处写得含糊，系统怎么处理？

提取阶段登记为歧义（分级：blocking / major / minor）。auto 模式下按最合理解释设为假设、高亮登记（major-auto），跑完统一在报告里呈现，每条附 revise 命令供你改口径重跑；interactive 模式下 blocking 级歧义直接挂起（paused_blocked），把候选解释与影响呈给你裁决后继续。

我有自己的回测框架，怎么让它用我的？

setup 第四项填你框架的目录路径（或编辑 .reproduce.json 的 backtest_framework）。复现规划与实现合同会收到指令：回测执行层优先调用你的框架、内置 common/ 只补缺口。输出产物格式与门禁要求不变，所以对数、审计、报告都照常。

怎么调整某个特定指标的容差，而不是全局容忍度？

直接编辑你目录里的 templates/standards.json——它在 setup 时作为种子拷贝到你目录，本来就是给你调的。按报告类型分节，每个指标可配 max_rel_dev（相对偏差上限）、abs_eps（近零绝对偏差）、require_same_sign（同号要求）、order_of_magnitude_only（仅量级）。全局偏差容忍与之的关系：非空时统一替换所有 max_rel_dev，其余语义保持。

流水线跑一半会话断了 / 想换台机器继续？

/reproduce continue <id>。全部进度在 workspace/{id}/state.json 与产物文件里，续跑时逐阶段幂等自愈：产物齐全的阶段直接跳过，缺的从断点重做。换机器把工作目录同步过去即可。

复现海外报告/论文，本地没有原文市场的数据怎么办？

用实验模式：起跑时加 --experimental。规划阶段会把原文不可得的数据映射为本地等价替代（如美国 CPI → 中国 CPI、SPX → 沪深300），每个替代都登记进假设登记簿；复现目标从「数值对齐原文」变为「该方法在迁移市场是否成立」——数值达标判定整体豁免（效应存在与否都是有效结论），但数字与产物的一致性照样被反虚报审计盯着（数字不能编）。最终报告与 HTML 结果页会强制显著声明这是市场迁移复现，门禁核验该声明缺失过不了关。若你没加 flag 而规划阶段发现原文市场整体不可得，系统会挂起询问你是否切换，不会擅自决定。

报告被分诊为 hard，跑得又慢又贵怎么办？

先看分诊依据（plan.md 的难度判定节）。v2.6.0 起判据以方法复杂度为主轴：只有训练模型 / 优化求解 / 多资产联动才 hard，结果表多、指标多属于"工作量大"，只影响里程碑数量不抬难度档。若仍觉得过重：起跑时 --difficulty medium 直接覆盖分诊；配置 economy: true 让机械性角色降为 sonnet；把 audit_level 调为 standard 让规格/代码外审只在关键条件下触发。三者叠加约可省三分之二 token 消耗。v2.7.0 又叠加了墙钟流水线（自动生效无需配置）：milestone 验证与下一段编码重叠、代码外审与运行验证默认并行、结果审计与样本外分析并发——medium 每跑再省约半小时。结果反虚报、达标门禁、人工闸门一处不动。

怎么更新插件？

/plugin update quant-report-reproduce（重启会话生效）。你目录里的 templates/common 不会被自动覆盖——setup 重跑时会列出「插件侧较新」的文件清单，由你决定是否手动同步。

维护者说明

面向插件维护者（仓库所有者）的分发与同步约定；普通使用者可跳过本节。

  • 分发一律用 git URL：marketplace add https://github.com/Hedicx2020/ghquant_plugin 只取 git 跟踪内容（约 4.6M）。marketplace add <本地路径> 会全量拷贝目录——含 .venv、output/ 等被 gitignore 的内容（实测 400M+），仅限本机自测。
  • 公开仓库是脱敏快照：远程 main 由 pub_snapshot 孤儿分支强推而来，不带开发历史，且剔除了验收案例文书（workspace/test_v2、src/test_v2）与研报原文 PDF。本地 main 保留完整历史，两者刻意不同源——不要在 main 上直接 git push。
  • 同步公开仓库的流程：切到 pub_snapshot → git checkout main -- . 铺最新内容 → git rm -r --cached 剔除脱敏项 → amend 重写快照提交 → 强推 origin main → 用 GitHub API 核查远程树零残留。（在维护者的 Claude Code 会话里说「同步公开仓库」即可自动执行。）
  • 路径定位机制：skill 正文不替换 ${CLAUDE_PLUGIN_ROOT}、Bash 也不注入该环境变量（已实测）——工具定位依赖 SKILL.md 协议第 0 级（skill base directory 上两级即插件根），不要在 skill/stages 中引入对该变量的新依赖。
  • 版本与更新：功能变更时同步 .claude-plugin/plugin.json 的 version；使用者 /plugin update quant-report-reproduce 后重启会话生效。用户目录的 templates/common 种子永不被自动覆盖。

术语速查

系统的文书（审计意见、门禁输出、覆盖矩阵）使用编号体系保证每条记录可追溯、可机器判定。日常使用不需要记它们——汇报都用人话，代号只在留痕文书里出现。查阅文书时对照下表：
